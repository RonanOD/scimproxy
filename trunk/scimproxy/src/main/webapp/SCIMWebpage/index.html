Overview

The System for Cross-domain Identity Management (SCIM) specification is designed to make managing user identity in cloud based applications and services easier. The specification suite seeks to build upon experience with existing schemas and deployments, placing specific emphasis on simplicity of development and integration, while applying existing authentication, authorization, and privacy models. It's intent is to reduce the cost and complexity of user management operations by providing a common user schema and extension model, as well as binding documents to provide patterns for exchanging this schema using standard protocols. In essence, make it fast, cheap, and easy to move users in to, out of, and around the cloud.

Information on this overview page is not normative.

Model

SCIM is build on a object model with inheritance the resource is the common denominator and all SCIM objects are derived from it. As can be seen in the image below SCIM has three object that directly inherits form the Resource object. The ServiceProviderConfiguration and Schema is used for discovery and contains no user information. The CoreResource is has where the user and group data and is contained within its two children User and Group.

Operations

For manipulation and of Resource SCIM provides a REST API with a rich but simple set of operation, everything from patching a specific attribute on a specific user to doing massive bulk updates.

  • Create = POST https://example.com/{v}/{resource}
  • Read = GET https://example.com/{v}/{resource}/{id}
  • Update = PUT https://example.com/{v}/{resource}/{id}
  • Delete = DELETE https://example.com/{v}/{resource}/{id}
  • Update = PATCH https://example.com/{v}/{resource}/{id}
  • Search = https://example.com/{v}/{resource}?ﬁlter={attribute}{op}{value}&sortBy={attributeName}&sortOrder={ascending|descending}
  • Bulk

Discovery

To simplify interoperability SCIM provides two end points form discovering of supported optional features and specific attribute details.

  • GET /ServiceProviderConﬁg

    Spec compliance, auth schemes, data models

  • GET /Schemas

    Introspect resources and attribute extensions

Create Request

Creating Resource is done with POSTs towards there respective end points. In the example below we see the creation of a User. As can be seen in this and later examples the URL contains a version number so that later versions can co-exists. available versions can be dynamically discovered with the ServiceProviderConﬁg end point. 

POST /v1/Users  HTTP/1.1
Host: example.com
Accept: application/json
Authorization: Bearer h480djs93hd8
{
  "schemas":["urn:scim:schemas:core:1.0"],
  "userName":"bjensen",
  "externalId":"bjensen",
  "name":{
    "formatted":"Ms. Barbara J Jensen III",
    "familyName":"Jensen",
    "givenName":"Barbara"
  }
}

Get Request

Fetching resources is done with HTTP get towards the desired Resource end point with the id in the path. In this example a user is fetched and at the end of the URL we can see the postfix '.json' this is for clients that cannot see the accept header to specify desired response format. 

GET /v1/Users/2819c223-7f76-453a-919d-413861904646.json  
Host: example.com
Accept: application/json
Authorization: Bearer h480djs93hd8

Filter Request

In addition to getting single Resources it is also possible to fetch sets of Resources by querying the Resource end point without the id for a specific user. To not have to read all users a filter can be applied to the resources. SCIM has support for filter operations equals, contains, starts with and more. In addition to filter the response it is also possible to ask the service provider to sort the Resources in the response.

  • https://example.com/{resource}?ﬁlter={attribute} {op} {value} & sortBy={attributeName}&sortOrder={ascending|descending}&attributes={attributes}

  • https://example.com/Users?ﬁlter=title pr and userType eq “Employee”&sortBy=title&sortOrder=ascending&attributes=title,username

Filter Response
